End to End · 端對端追蹤

實際寫一筆資料：
從鍵盤到硬碟

具體場景：你在某個社群網站發文「今天天氣真好」，按下「發布」。 下面是這個文字字串走過的所有環節。

1

瀏覽器層：捕捉與封裝

JavaScript 在前端做的事

React 元件捕捉 textarea 的 onChange 事件，把文字存到 React state。 按下「發布」按鈕，元件呼叫 onClick handler，做前端驗證（不能空、不超過字數）， 然後用 fetch() 把資料包成 JSON 送出。

await fetch('/api/posts', {
  method: 'POST',
  headers: { 'Authorization': `Bearer ${jwt}` },
  body: JSON.stringify({ content: '今天天氣真好' })
});
2

網路層：加密、路由

瀏覽器把這個 POST 請求用 TLS 加密，透過家中網路 → ISP → 國際線路 → 目的地機房， 一路上會經過很多路由器。中途如果有 CDN，會先到最近的邊緣節點，再轉發到原始伺服器。

3

邊緣層：CDN / WAF 過濾

CDN 邊緣節點先看這個請求：是惡意嗎？（WAFWeb Application Firewall 過濾） 是 DDoS 嗎？頻率正常嗎？通過了才會轉發到後端。

4

後端入口：Load Balancer

到達機房後，Load Balancer 把這個請求分派給目前最閒的應用伺服器。同時做 SSL 終止—— 這裡開始內部走 HTTP（內網本來就安全，省去加密成本）。

5

API Gateway：身份驗證、路由

驗證 JWT 有效、解析出使用者身份、查 rate limit 沒超過，然後根據網址路由到正確的微服務。 /api/posts 對應 Post Service，把請求轉發過去。

6

微服務：業務邏輯

Post Service 收到請求，做後端驗證（不要相信前端送來的資料）、 檢查使用者是否有發文權限、過濾敏感字、產生 ID 與時間戳，然後準備寫入資料庫。

7

資料庫：寫入持久化儲存

ORM 把物件轉成 SQL 語句送給 PostgreSQL。資料庫先寫進WAL（Write-Ahead Log）確保不會丟失， 然後更新 B-Tree 索引、寫入 table。最終資料躺在實體硬碟的特定區塊。

INSERT INTO posts (user_id, content, created_at)
VALUES (123, '今天天氣真好', NOW())
RETURNING id;

# 同步副作用
- 寫入 WAL（保證 crash 後可恢復）
- 更新索引（user_id, created_at）
- 寫入 disk block
- 同步到 replica（若有設定）
8

非同步副作用：訊息佇列

主流程已經結束，但還有些事要做：通知粉絲、更新搜尋索引、計算熱門排行。 Post Service 把事件 post.created 丟進 Kafka，多個下游服務各自訂閱：

背景處理者
  • Notification Service推播給粉絲
  • Search Service更新 Elasticsearch 索引
  • Analytics Service記錄發文事件用於統計
  • Trending Service更新熱門排行榜
9

回應路徑：原路返回

Post Service 回傳新建的 post 物件給 API Gateway → Load Balancer → 邊緣節點 → 網際網路 → 你的瀏覽器。 前端拿到回應後，更新本地 state，React 重新 render，畫面上多了一篇你的新貼文。

整個過程通常在 100~300ms 內完成，雖然涉及十幾個系統。

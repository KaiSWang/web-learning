Option C · The Backend Side

請求進入伺服器後，
到底經過了哪些手？

Vol.02 第一幕我們追到「請求送到 Google 伺服器」就停了。但「伺服器」其實不是一台機器—— 它是一整套分工合作的系統。請求會經過負載均衡、反向代理、應用伺服器、資料庫、快取、訊息佇列等各種角色， 最後才能組出回應。這一章我們從後端視角看完整鏈路。

Architecture · 後端架構全景

一個請求進來，
多少個角色會接力處理？

下面這張圖是現代後端的典型架構。從外部進來的請求，會被多層處理後才到達真正寫業務邏輯的程式碼。 每一層都有特定職責。

▸ 後端服務分層架構

每個方框都可能由多台機器組成。Google 這種規模的服務每層都是上千台。

Layer by Layer · 逐層拆解

每一層都解決什麼問題？

1

Load Balancer · 負載均衡器

分流的交通警察

一個服務如果只有一台伺服器，那台壞了整個網站就掛了。所以實際部署時會有多台一模一樣的應用伺服器， 前面放一個 Load Balancer 把流量平均分給它們。這也讓系統能水平擴展——流量大時多開幾台就好。

分流策略
  • Round Robin輪流分配，最簡單
  • Least Connections分給目前連線數最少的
  • IP Hash同一個使用者每次都打到同一台（保持狀態用）
  • Health Check定期檢查每台是否健康，壞掉的自動踢掉

常見的 Load Balancer：Nginx、HAProxy（自己架）、AWS ELB、Cloudflare Load Balancer（雲端託管）。 你之前 sino-suisse 用過的 ELB 就是 AWS 的負載均衡器。

2

Reverse Proxy · 反向代理

應用前的守門員

通常和 Load Balancer 是同一台機器（Nginx 同時做這兩件事）。它做的事包括： SSL 終止（HTTPS 在這裡解密，後面內部都走 HTTP）、 靜態檔案直接回（圖片、CSS 不用麻煩應用伺服器）、 請求改寫、Rate Limiting（防止 DDoS）。

3

Application Server · 應用伺服器

真正執行業務邏輯的地方

這是後端工程師寫程式的地方——Node.js、Python、Go、Java、Ruby 等都是常見選擇。 收到請求後，這層會：解析請求 → 驗證身份 → 路由到對應的 Controller → 執行業務邏輯 → 組裝回應。

# 一個簡化的請求處理流程（以 Express.js 為例）
app.get('/users/:id', async (req, res) => {
  // 1. 驗證 JWT
  const user = await verifyToken(req.headers.authorization);
  
  // 2. 查快取
  const cached = await redis.get(`user:${req.params.id}`);
  if (cached) return res.json(JSON.parse(cached));
  
  // 3. 查資料庫
  const data = await db.users.findById(req.params.id);
  
  // 4. 寫入快取
  await redis.setex(`user:${req.params.id}`, 60, JSON.stringify(data));
  
  // 5. 回傳
  res.json(data);
});
4

Cache · 快取層

記憶體存的「常見答案」

資料庫查詢很慢（毫秒等級），記憶體快取很快（微秒等級）。把常被問到的資料放在記憶體裡， 可以讓系統承受 10 倍以上的流量。 最常用的快取是 Redis（功能多、可持久化）和 Memcached（純粹快、簡單）。

快取的常見模式
  • Cache-Aside應用自己管，沒命中就查資料庫並寫入快取（最常見）
  • Read-Through快取自己負責同步，應用永遠查快取
  • Write-Through寫入時同時寫快取和資料庫
  • Write-Behind先寫快取，背景慢慢同步到資料庫
  • TTL每筆快取設過期時間，避免永遠陳舊
5

Database · 資料庫

事實的最終來源

資料的「家」。分兩大類：SQL（關聯式）和 NoSQL（非關聯式）。

類型 代表 特性 適合場景
關聯式 SQL PostgreSQL, MySQL 嚴格 schema、強一致性、JOIN 需要交易的系統（金融、訂單）
文件型 NoSQL MongoDB JSON 結構、彈性 schema 內容管理、Log 分析
鍵值型 NoSQL Redis, DynamoDB 極快、純 key-value 快取、Session、計數器
欄位型 Cassandra, BigQuery 大量寫入、分析查詢 資料倉儲、IoT 數據
圖形型 Neo4j 處理關聯 社群網路、推薦系統
6

ORM · 物件關聯映射

資料庫的翻譯官

ORMObject-Relational Mapping 讓你不用寫 SQL，用程式語言的物件操作就能讀寫資料庫。 它在背後幫你產生 SQL、處理連線池、做型別轉換。

# 沒有 ORM（直接寫 SQL）
db.query("SELECT * FROM users WHERE id = ?", [123]);

# 有 ORM（更直觀）
User.findOne({ id: 123 });
主流 ORM
  • Prisma現代 TypeScript ORM，型別安全
  • SQLAlchemyPython 主流 ORM
  • GORMGo 主流 ORM
  • HibernateJava 老牌 ORM
7

Message Queue · 訊息佇列

非同步任務的傳送帶

有些任務很慢（寄信要幾秒、產生報表要幾分鐘），不能讓使用者乾等。 這時應用伺服器只要把任務「丟到佇列」，就馬上回應使用者「已收到」， 背景的 Worker 慢慢從佇列裡拿任務來執行。

# 同步做法：使用者要等 5 秒
[使用者點「下單」] ─▸ 處理訂單 ─▸ 寄確認信(5秒) ─▸ 回應

# 非同步做法：使用者立刻拿到回應
[使用者點「下單」] ─▸ 處理訂單 ─▸ 丟訊息進 Queue ─▸ 立即回應
                                         Worker ─▸ 寄信（背景處理）
主流訊息佇列
  • RabbitMQ傳統訊息佇列，支援多種模式
  • Apache Kafka高吞吐量、串流處理
  • AWS SQS雲端託管
  • Redis Streams輕量級選擇

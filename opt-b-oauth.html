Option B · OAuth 2.0 Deep Dive

「用 Google 登入」
按鈕背後的完整流程

你在很多網站看過「使用 Google 帳號登入」「使用 Facebook 登入」按鈕。點下去之後，你的密碼從來沒給過那個網站， 但它居然成功讓你登入了——這個「不用給密碼也能授權」的魔法，就是 OAuth 2.0。

The Problem · 為什麼需要 OAuth

沒有 OAuth 之前，
世界長什麼糟糕的樣子？

想像 2010 年，有個第三方網站想幫你管理 Gmail 信件。 它會跟你要 Google 帳號和密碼，存在自己的資料庫裡，每次幫你查信時用你的密碼登入 Gmail—— 這有三個致命問題。

The Solution · OAuth 2.0 流程

OAuth 怎麼解決
這些問題？

核心想法：第三方永遠不接觸你的密碼，而是去 Google 拿一張「授權票券」（Token）， 憑票券去做有限的事情，票券可以隨時撤銷。

完整 14 個步驟。看似複雜，但每一步都在解決一個安全問題。

Key Concepts · 關鍵概念

OAuth 流程中的四個主角

A

Authorization Code · 授權碼

短命的「臨時通行證」

Google 在使用者同意授權後，產生一個短暫有效的 Code（通常 10 分鐘）。 這個 Code 會透過瀏覽器網址傳給第三方 App。

為什麼不直接給 Access Token？因為網址容易被偷看（瀏覽器歷史、伺服器日誌）。 授權碼即使被偷，沒有 client_secret 也換不到 Token。

B

Access Token · 存取權杖

真正的「鑰匙」

憑這個 Token 才能呼叫 Google 的 API（拿 Email、行事曆、雲端硬碟資料）。 通常1 小時過期，過期後要用 Refresh Token 換新的。

為什麼設這麼短？因為一旦洩漏，攻擊者只能用 1 小時，傷害有限。

C

Refresh Token · 刷新權杖

「續約」用的長期憑證

這個 Token 通常數週或永久有效，但只能用來換新的 Access Token，不能直接呼叫 API。 存放在後端伺服器（不放前端）。

D

Scope · 權限範圍

細粒度授權

App 一開始要求授權時，必須宣告自己想存取什麼。例如： scope=email profile（只要基本資料）vs scope=gmail.readonly（只能讀信，不能寫信）。 使用者授權時看得到這份清單，可以選擇拒絕。

常見 Scope 範例
  • openidOpenID Connect 標準的基本識別
  • email取得 email 地址
  • profile取得姓名、頭像
  • gmail.readonly只能讀 Gmail 信件
  • drive.file只能存取 App 自己建立的檔案
Bonus · JWT 是什麼

JWT vs Session：
兩種身份管理方式

OAuth 常與 JWT（JSON Web Token）一起出現。它是一種特殊的 Token 格式—— 跟傳統 Session 比起來有重要的架構差異。

面向 Session（傳統） JWT（現代）
狀態存哪 伺服器資料庫（記錄誰有效） Token 本身（自包含資訊）
每次驗證 查資料庫 驗簽章（CPU 運算）
分散式擴展 難，Session 要共享 易，無狀態
強制登出 直接刪 Session 難，要等過期或維護黑名單
Token 大小 小（只是 ID） 大（含資訊）
適合場景 單一網站、傳統 Web App 微服務、跨服務、行動 App
# JWT 長這樣（用 . 分三段）
eyJhbGciOiJIUzI1NiIs... # Header（演算法資訊）
.
eyJzdWIiOiIxMjM0... # Payload（使用者資訊）
.
SflKxwRJSMeKKF2QT4... # Signature（簽章驗證）

# 解碼後的 Payload 範例
{
  "sub": "user_123",
  "name": "KS",
  "exp": 1730000000 # 過期時間
}

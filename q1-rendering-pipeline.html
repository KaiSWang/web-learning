60fps 的祕密：
瀏覽器一秒能畫多少次？

Option A 我們看了渲染管線的六個步驟。這一章鑽進實際運作層面： 為什麼網頁要追求「60fps」？瀏覽器引擎內部如何安排這些工作？ 畫面卡頓究竟是哪個環節的問題？

Frame Budget · 一個畫格的預算

每個畫格只有16.67 毫秒

螢幕通常是 60Hz 更新率，意思是每秒重新顯示 60 次。所以每一幀（frame）只有 1000ms ÷ 60 = 16.67ms 的時間預算。 在這短短時間內，瀏覽器要做完 JS 執行、樣式計算、Layout、Paint、Composite——任何一步超時就會「掉幀」。

▸ 一個畫格的時間軸（16.67ms）

JS 執行佔太久 → 後面所有階段被延後 → 視覺上感覺到「滑動卡卡的」

為什麼追求 60fps？
  • 30fps勉強流暢，影像可接受，但互動感不佳
  • 60fps絲滑感，主流標準
  • 120fps新款 iPhone、iPad、ProMotion 螢幕（每幀只有 8.33ms）
  • 低於 24fps明顯卡頓感
Engine Internals · 瀏覽器引擎內部

瀏覽器其實是
很多執行緒在合作

你以為「瀏覽器」是一支程式？實際上現代瀏覽器是多進程多執行緒架構， 每個 Tab 是獨立進程，每個進程內又有多個執行緒分工。

▸ Chromium 多進程架構

這就是為什麼 Chrome 開很多分頁吃很多 RAM——每個都是獨立進程

Optimization · 維持 60fps 的技巧

實戰：怎麼讓網頁不掉幀？

A

動畫只用 transform 和 opacity

這兩個屬性跳過 Layout 和 Paint，直接交給 Compositor Thread。 其他屬性（如 left、width）會觸發整條管線重跑。

# ❌ 慢：觸發 Layout
element.style.left = '200px';

# ✓ 快：只觸發 Composite
element.style.transform = 'translateX(200px)';
B

Web Worker：把重活搬出主執行緒

主執行緒一旦卡住，使用者就感覺得到。把長時間運算（解析大 JSON、加密、影像處理）丟到 Web Worker， 它在背景執行緒跑，不影響畫面。

# 主執行緒
const worker = new Worker('heavy.js');
worker.postMessage(largeData);
worker.onmessage = (e) => updateUI(e.data);

# heavy.js 在背景跑，主執行緒繼續處理使用者互動
C

requestAnimationFrame：跟著畫面同步

想做 JS 動畫？別用 setInterval。requestAnimationFrame 會在每幀開始時呼叫你的回調， 自動配合螢幕更新率（60Hz / 120Hz 自動適應）。

D

避免 Layout Thrashing（Layout 抖動）

JS 中反覆「讀 layout 屬性 → 寫 layout 屬性 → 讀 → 寫」會觸發強制同步 layout，效能極差。 正確做法是批次讀完再批次寫。

# ❌ 抖動：每次迴圈都觸發 layout
for (...) {
  const h = element.offsetHeight; // 讀
  element.style.height = h + 10 + 'px'; // 寫
}

# ✓ 批次：先讀完再寫
const heights = elements.map(el => el.offsetHeight);
elements.forEach((el, i) => el.style.height = heights[i] + 10 + 'px');
E

Virtual List：只 render 看得到的東西

列表有 1 萬筆資料？實際 DOM 上不需要 1 萬個節點。 「虛擬列表」只渲染目前在視窗內的 20 筆，捲動時動態替換內容。 這是 LINE 聊天列表、IG 動態的底層技術。

跨國網站
為什麼還能這麼快？

你在台北打開美國的新聞網站，幾乎沒延遲。但物理上資料要走過太平洋海底電纜——光速也要 100ms。 網站怎麼克服這個距離？答案是 CDN（Content Delivery Network）， 它把內容「複製」到世界各地，讓你不用真的連到美國。你之前處理 sino-suisse.com 的 Cloudflare 遷移，就是在玩這個系統。

Architecture · CDN 全球架構

CDN 怎麼把內容
放到你身邊？

傳統做法：所有使用者都連到原始伺服器（origin server），距離越遠越慢。 CDN 做法：在全球幾百個城市建邊緣節點（PoP, Point of Presence），原始內容快取在最近的節點。

▸ 沒 CDN vs 有 CDN

Cloudflare 在全球有 300+ 城市的 PoP，平均到使用者距離 50ms

Caching · 快取策略

CDN 怎麼知道該不該回快取？

CDN 不是無腦快取所有東西。哪些可以快取多久、什麼時候要刷新——這些靠 HTTP Header 來控制。 理解這些對你之前處理 sino-suisse 遷移很有幫助。

A

Cache-Control · 主要的快取指令

最重要的 Header。告訴 CDN 和瀏覽器這份內容能快取多久、誰可以快取。

# 公開內容，CDN 和瀏覽器都快取一年
Cache-Control: public, max-age=31536000, immutable

# 只允許瀏覽器快取（不快取在 CDN）
Cache-Control: private, max-age=3600

# 完全不快取（每次都要重新請求）
Cache-Control: no-store

# 可快取，但每次使用前都要驗證
Cache-Control: no-cache
B

ETag · 內容指紋驗證

伺服器給每份內容一個「指紋」（通常是內容的 hash）。瀏覽器下次來問時，帶上指紋： 「我有版本 abc，還是最新嗎？」如果是，伺服器只回 304 Not Modified（沒 body，省頻寬）。

# 第一次請求
▸ GET /image.png
◂ 200 OK + ETag: "abc123" + 圖片內容

# 第二次請求（快取已過期但內容沒變）
▸ GET /image.png
  If-None-Match: "abc123"
◂ 304 Not Modified (沒 body，省 99% 頻寬)
C

Cache Key · 怎麼判斷是「同一個東西」

預設用 URL 當 key，但有些情況需要更細的區分。例如同一個網址但不同語言、不同裝置要看不同內容。 Vary: Accept-Language 告訴 CDN「依語言分別快取」。

D

Cache Purge · 強制刷新

內容更新後，怎麼讓 CDN 上的舊版立刻失效？大部分 CDN 提供 Purge API， 可以指定 URL 或 tag 立即清除。Cloudflare 還支援 Cache Tag—— 一次清掉所有相關內容。

Edge Computing · 邊緣運算

從 CDN 到
邊緣執行程式

CDN 一開始只是快取靜態內容。後來人們想：既然全球都有節點，何不讓程式碼也在邊緣執行？ 這就是 Edge Computing。

▸ 三種運算位置比較

三層各有所長，現代架構會混搭使用

Edge 適合做的事
  • A/B Testing在邊緣決定使用者看哪個版本
  • 身份驗證JWT 在邊緣驗證，無效直接擋掉
  • 圖片即時處理邊緣節點動態壓縮、轉檔
  • 地理路由依使用者位置導到不同 origin
  • Bot 偵測過濾爬蟲、惡意流量
主流邊緣平台
  • Cloudflare Workers用 V8 isolate，啟動 0ms（你最熟悉的）
  • AWS Lambda@EdgeAWS 的邊緣 Lambda
  • Vercel Edge FunctionsNext.js 整合
  • Deno Deploy原生支援 TypeScript

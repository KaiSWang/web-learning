第一幕 · The Connection

你輸入網址後，
瀏覽器怎麼找到 Google？

你輸入的是「google.com」，但網路世界其實只認 IP 位址（像 142.250.196.78）。 所以瀏覽器要做的第一件事，就是把名字翻譯成地址。

1.1

查地址簿：DNS 解析

DNSDomain Name System · 網域名稱系統 網域名稱系統

瀏覽器先問你的電腦：「google.com 的 IP 是多少？」電腦如果不知道，就去問你家路由器；路由器不知道，就問 ISPInternet Service Provider（中華電信、Hinet）的 DNS 伺服器； 一路問下去直到找到答案。這整個過程通常10 毫秒內完成，而且結果會被快取，下次更快。

▸ DNS 查詢層級

逐層問，找到答案後沿路快取，下次同網域查詢只需 0~1ms

這層牽涉的概念
  • DNS把網域名稱翻譯成 IP 位址
  • Cache查過的會暫存，避免每次都重查
  • TTLTime-To-Live，快取保留多久（你之前處理 sino-suisse.com CDN 遷移時碰過）
  • Record TypeA 記錄（指向 IP）、CNAME（指向別的網域）、MX（郵件伺服器）等
1.2

建立連線：TCP 三次握手 + TLS 加密

TCP / TLS Handshake

拿到 IP 後，瀏覽器跟 Google 伺服器「打招呼」建立連線。這像兩個人通話前要先確認「你聽得到嗎？聽得到，你呢？我也聽得到」—— 這就是 TCPTransmission Control Protocol 三次握手。 確認連得上之後，再做一次TLSTransport Layer Security 加密協商， 雙方交換金鑰，之後所有對話都會被加密——這就是網址前面那個 https:// 的鎖頭由來。

# TCP 三次握手
瀏覽器 ─SYN─▸ 伺服器   「我要連你」
瀏覽器 ◂SYN+ACK─ 伺服器  「OK，你也要回應我」
瀏覽器 ─ACK─▸ 伺服器   「收到，連線建立」

# TLS 加密協商（簡化版）
瀏覽器 ─▸ Hello + 支援的加密套件 ─▸ 伺服器
伺服器 ─▸ 選定的套件 + SSL 憑證 ─▸ 瀏覽器
雙方 ─▸ 交換金鑰 ─▸ 後續通訊全部加密 🔒
這層牽涉的概念
  • TCP確保資料能可靠送達的傳輸協定
  • HTTPSHTTP + TLS = 加密過的網頁協定
  • SSL 憑證由公信第三方（如 Let's Encrypt）簽發，證明「我真的是 google.com 不是釣魚網站」
  • HTTP/2、HTTP/3更新更快的版本（HTTP/3 用 QUIC 協定，基於 UDP）
1.3

送出請求：HTTP Request

The Request

連線建好了，瀏覽器送出一個請求，告訴 Google 伺服器： 「我要拿 / 這個路徑的內容，我用的是 Chrome 瀏覽器，我接受 zh-TW 語言…」 這份請求由「方法（GET）+ 路徑 + 標頭（Headers）+ 內容（Body）」組成。

GET / HTTP/2
Host: google.com
User-Agent: Chrome/120 (Macintosh)
Accept-Language: zh-TW
Cookie: SID=xxxxx... ← 這個 Cookie 是登入的關鍵！
1.4

伺服器回應：HTTP Response

The Response

Google 的伺服器收到請求，根據你的 Cookie 知道「啊這是 KS」， 就準備一份專屬於你的 HTML 文件送回來（已經帶有你的頭像、最近搜尋紀錄等）。 回應裡含 HTML、CSS、JavaScript、圖片等資源，瀏覽器收到後開始解析、渲染成畫面。

這層牽涉的概念
  • Status Code200 OK / 301 重新導向 / 404 找不到 / 500 伺服器錯誤
  • HTMLHyperText Markup Language，網頁的結構（骨架）
  • CSSCascading Style Sheets，樣式（皮膚）
  • JS / JavaScript互動邏輯（神經系統）
  • CDNContent Delivery Network，把這些資源放在離你最近的伺服器（你熟悉的 Cloudflare）
第二幕 · The Identity

為什麼不用每次登入？
跨服務又怎麼共用帳號？

HTTP 其實是無狀態的協定——伺服器每收到一次請求都不認識你。 那為什麼你關掉瀏覽器明天打開還是登入狀態？這靠的是 Cookie + Session + Token 這套機制。

▸ Cookie 在跨子網域的共用機制

Cookie 的 Domain 設定為 .google.com，讓所有子網域共用登入狀態（SSO 的基礎）

2.1

第一次登入：拿到一張「通行證」

Session & Cookie

你第一次輸入帳號密碼，伺服器驗證通過後，會做兩件事： ① 在伺服器端記下「KS 這個人有效，編號 abc123」； ② 把編號 abc123 寫進你瀏覽器的 Cookie 裡。 之後每次發送請求，瀏覽器都會自動把這個 Cookie 帶上去——伺服器一看就知道：「喔是 KS」。

# 登入後，伺服器回傳的 Header 裡會有：
Set-Cookie: SID=abc123; Domain=.google.com;
HttpOnly; Secure; Max-Age=2592000

# 之後每次請求 google.com 旗下任何網站
# 瀏覽器都會自動帶上這個 SID
2.2

關鍵在 Domain：為什麼 Gmail、Drive 都認得你？

SSOSingle Sign-On 單一登入

Cookie 設定時可以指定 Domain=.google.com， 意思是「凡是 *.google.com 結尾的網站都共用這個 Cookie」。 所以你登入 google.com 之後，瀏覽器去 mail.google.com、drive.google.com， 這個 Cookie 都會被帶過去——這就是單一登入（SSO）的基礎。

這層牽涉的概念
  • Cookie Domain決定哪些子網域共用 Cookie
  • SSOSingle Sign-On，登入一次多服務通用
  • OAuth 2.0跨網域授權的標準協定（「用 Google 登入」的原理，詳見 Option B）
  • JWTJSON Web Token，另一種身份識別格式，把資訊直接編碼在 Token 裡
2.3

關掉瀏覽器再開，為什麼還登入著？

Persistent Cookie

Cookie 有兩種：Session Cookie（關閉瀏覽器就消失）和 Persistent Cookie（指定有效期，存在硬碟裡）。 Google 給你的是後者，Max-Age=2592000 代表 30 天有效。 所以你關機、重開、隔天再登入，Cookie 還在，伺服器看到 SID 還沒過期就直接認得你。

2.4

安全性：為什麼 Cookie 不會被偷？

Security Flags

Cookie 上會帶幾個安全標記：

Cookie 的安全護欄
  • HttpOnlyJavaScript 讀不到，防止 XSS 攻擊偷 Cookie
  • Secure只在 HTTPS 連線傳送，不走明文
  • SameSite限制跨站請求帶 Cookie，防 CSRF 攻擊
  • CSPContent Security Policy，限制可載入的資源來源（防止注入惡意腳本）
第三幕 · The Navigation

為什麼點按鈕
畫面就瞬間切換？

早期網站每次點連結，整個頁面會重新載入（白一下再出現）。現代網站像 Gmail、Notion 點什麼幾乎都不重整，畫面切得超快——這背後是兩種完全不同的架構思路。

▸ MPA vs SPA 架構對比

三種架構並存於現代網頁世界，各有適用場景

3.1

傳統做法：每次點擊重新載入整頁

MPAMulti-Page Application 多頁式應用

MPA（Multi-Page Application）的意思是「多頁式應用」——網站由很多獨立 HTML 頁面組成。 點擊連結 → 瀏覽器發新請求 → 伺服器回傳一份新的 HTML → 整個頁面重新渲染。 缺點是慢、有白屏，但好處是架構單純、SEOSearch Engine Optimization（搜尋引擎優化）友善。 維基百科、新聞網站多半是這種架構。

為什麼 SEO 友善？因為 Google 爬蟲打開每個 URL 都直接拿到完整的 HTML 內容，能讀懂頁面在講什麼。 SPA 早期被批評「Google 看不懂」就是這個原因。

3.2

現代做法：只換中間那塊

SPASingle-Page Application 單頁應用

SPA（Single-Page Application）的意思是「單頁應用」——整個網站其實只是一張 HTML 頁面，所有切換都在這張頁上完成。 首次載入時就把整個 App 的「框架」抓下來（HTML + JS + CSS）。之後你點任何按鈕， JavaScript 攔截點擊事件，只去後端要「需要更新的那塊資料」（透過 API）， 拿回來後動態替換 DOM 裡的某個區塊——網址也用 history.pushState() 換掉， 看起來像跳頁，其實沒重新載入。

這層牽涉的概念
  • DOMDocument Object Model，瀏覽器把 HTML 解析成的物件樹，JS 操作的對象
  • APIApplication Programming Interface，後端開放給前端呼叫的資料介面
  • AJAXAsynchronous JavaScript and XML，老派但常用的非同步請求技術
  • Fetch現代版的 AJAX，內建在瀏覽器
  • JSONJavaScript Object Notation，API 之間傳資料的格式
  • React / Vue幫你管理「資料變了 → DOM 怎麼更新」的框架
  • Router處理「網址變了但不重整」的邏輯
3.3

第三條路：兩者混合

SSR / SSG / Hydration

SPA 雖然快，但首次載入很重，而且搜尋引擎抓不到內容。 所以現代框架（Next.js、Nuxt）走混合路線：首頁由伺服器先渲染好 HTML 送來（快、SEO 好）， 瀏覽器收到後再「啟動」JavaScript 讓它變成 SPA（後續切換快）。這個過程叫 Hydration（水合）。

這層牽涉的概念
  • SSRServer-Side Rendering，伺服器端先渲染（每次請求即時產生）
  • SSGStatic Site Generation，建置時就產生靜態 HTML（最快）
  • CSRClient-Side Rendering，瀏覽器端渲染（傳統 SPA 模式）
  • Hydration讓伺服器渲染的 HTML「活過來」變成可互動的 SPA
  • Next.js / Nuxt / SvelteKit主流的混合架構框架

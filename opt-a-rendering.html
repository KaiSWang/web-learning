Option A · Browser Rendering Pipeline

瀏覽器拿到 HTML，
怎麼變成你眼前的畫面？

Vol.02 第三幕說「JS 攔截點擊、換 DOM、畫面更新」——但「換 DOM 之後畫面更新」這句話跳過了非常多步驟。 這一章我們鑽進瀏覽器內部，看它怎麼把一段純文字（HTML/CSS）變成你看到的精美畫面， 以及這條管線在哪些地方會卡住、為什麼有些網頁會 lag。

The Pipeline · 渲染管線總覽

瀏覽器內部的六道工序

從拿到一份 HTML 字串，到畫面出現在螢幕上，瀏覽器要經過六個階段。 這條管線在每一秒會跑很多次（為了維持 60fps，每秒要跑 60 次）。

六道工序組成 Critical Rendering Path（關鍵渲染路徑）

Step by Step · 每一步詳解

每一道工序裡，
到底發生了什麼？

①

Parse · 把 HTML 變成 DOM 樹

HTML String → DOM Tree

HTML 對瀏覽器來說只是一串字符。第一步是把這串字符「斷詞」（Tokenize）成有意義的標籤， 再根據標籤的巢狀關係建出一棵樹狀結構——這就是 DOM。

# 原始 HTML
<html>
  <body>
    <h1>Hello</h1>
    <p>World</p>
  </body>
</html>

# 解析後的 DOM 樹
html
 └─ body
    ├─ h1 ─ "Hello"
    └─ p ─ "World"

DOM 不只是抽象結構，每個節點都是 JavaScript 可以操作的物件。當你寫 document.querySelector('h1')， 你就是在這棵樹上找節點。

②

Style · 把 CSS 變成 CSSOM

CSS → CSSOM (CSS Object Model)

同時間，瀏覽器也在解析 CSS，建出另一棵樹叫 CSSOM（CSS Object Model）。 它記錄哪些選擇器、套用什麼樣式、優先級多高。

# CSS
body { font-size: 16px; }
h1 { color: red; font-size: 32px; }

# CSSOM
body { font-size: 16px }
  └─ h1 { color: red, font-size: 32px }
# 注意：h1 也繼承了 body 的設定，但有自己的覆寫

CSS 是「render-blocking」資源——瀏覽器拿不到完整的 CSSOM 之前不能繼續， 這就是為什麼 CSS 檔案要放在 <head> 裡盡早載入。

③

Combine · 合併成 Render Tree

DOM + CSSOM = Render Tree

把 DOM 和 CSSOM 結合，產生「實際要顯示的東西」這棵樹——稱為 Render Tree。 這裡會剔除不會顯示的節點（如 display: none 或 <head> 裡的東西）。

④

Layout · 計算位置與尺寸

又叫 Reflow（回流）

Render Tree 知道「要顯示什麼」，但不知道「在哪裡、多大」。Layout 階段就在算這個。 從根節點開始往下，計算每個元素的精確座標 (x, y) 和尺寸 (width, height)。

這是最昂貴的一步。任何讓「位置或尺寸改變」的操作都會觸發 reflow， 而且常常是連鎖反應——改一個元素，旁邊和下面所有元素都要重算。

# 觸發 Layout 的常見動作（昂貴）
element.style.width = "200px";
element.style.padding = "20px";
element.classList.add("expanded");
window.resize;

# 不觸發 Layout 的動作（便宜）
element.style.color = "red";
element.style.background = "blue";
⑤

Paint · 繪製像素

把每個元素畫出來

知道位置和尺寸後，把每個元素的「外觀」繪製出來——填色、邊框、文字、陰影、漸層。 這個階段瀏覽器會把畫面分成很多圖層（Layer），不同圖層分開繪製。

⑥

Composite · 圖層合成

GPU 加速的關鍵

把所有圖層按正確順序疊起來，這個工作會交給 GPU 處理（GPU 就是擅長這種大量平行運算的硬體）。 這是為什麼 CSS 動畫如果只改 transform 和 opacity 會特別順—— 因為它們只觸發 Composite，不需要重跑前面的 Layout 和 Paint。

效能優化的核心知識
  • transform: translate移動元素，只觸發 Composite，順
  • opacity透明度，只觸發 Composite，順
  • top / left移動元素，會觸發 Layout，慢
  • will-change告訴瀏覽器「這個元素即將改變」，提早分層
  • transform: translateZ(0)強制建立新圖層的小技巧
Performance · 為什麼會卡？

網頁卡頓
到底卡在哪一道？

理解了管線，你就能診斷卡頓。卡頓有不同等級，每個等級的解法也不同。

卡頓類型 觸發原因 每秒成本 解法
Layout 抖動 JS 反覆讀寫 layout 屬性（如 offsetTop） 高 批次讀取後再批次寫入
大量 Paint 動畫改 background、box-shadow 中 改用 transform / opacity
JS 主執行緒卡住 長時間運算（迴圈、解析大 JSON） 高 Web Worker 移到背景執行緒
圖層過多 濫用 will-change 或 translateZ 中 只在真正需要動畫的元素分層

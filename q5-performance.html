Q · 05 · Performance Diagnosis

網站慢，
到底慢在哪一步？

使用者只感覺「這網站好慢」，但工程師需要精確定位：是 DNS 慢？伺服器慢？資源太大？JS 執行卡？ Google 為了量化這件事，定義了 Core Web Vitals——三個關鍵指標。

Core Web Vitals · 三大核心指標

Google 怎麼量化網站速度？

Google 從 2020 開始強推 Core Web Vitals，把使用者體驗拆成三個維度： 載入速度（LCP）、互動回應（INP）、視覺穩定（CLS）。這些指標也會影響 SEO 排名。

Common Causes · 常見的慢點

網頁卡頓的典型原因

1

TTFB 太久 · 伺服器慢

Time To First Byte

從發出請求到收到第一個 byte 的時間。如果這段太久，後面再怎麼優化都沒用。 常見原因：資料庫查詢慢、伺服器離使用者遠、沒用 CDN、伺服器負載高。

改善方向
  • 用 CDN讓使用者連到最近的節點
  • SSG / Edge把渲染移到邊緣或建置期
  • 資料庫優化加索引、查詢快取
  • 水平擴展多開幾台應用伺服器
2

資源太大 · 圖片、JS、字體

首頁 HTML 才幾 KB，但圖片動輒幾 MB。一張未壓縮的相機原圖可能 10MB+， 手機 4G 要下載好幾秒。JS bundle 也經常臃腫——一個 React 網站 main.js 動輒 500KB+。

改善方向
  • 圖片格式用 WebP / AVIF 取代 JPG/PNG（同畫質減 30%~80%）
  • Lazy Loading滑到才載入（<img loading="lazy">）
  • Code SplittingJS 拆成小塊，按需載入
  • Tree Shaking移除沒用到的程式碼
  • 字體子集化只載入網頁實際用到的字
3

渲染阻塞 · CSS 和 JS 擋路

<link rel="stylesheet"> 和 <script> 預設會阻塞渲染——瀏覽器要等它們下載完才能繼續。網路慢時這影響很大。

# 阻塞渲染
<link rel="stylesheet" href="big.css">
<script src="big.js"></script>

# 不阻塞
<link rel="preload" as="style" href="big.css">
<script src="big.js" defer></script>
<script src="analytics.js" async></script>
4

主執行緒阻塞 · INP 變差

JS 執行佔用主執行緒太久，使用者點按鈕沒反應（因為主執行緒還在跑前一個任務）。 這在大型 React 應用很常見。解法見 Q1 的「Web Worker、避免 Layout Thrashing」。

5

CLS · 元素亂跳

最常見原因：圖片沒設定尺寸。HTML 載入時不知道圖片多高，先空著， 等圖片下載完成才把空間「擠出來」，導致下方所有內容往下跳——使用者剛要點某個按鈕，按鈕跳走了。

# ❌ 沒設尺寸
<img src="photo.jpg">

# ✓ 設尺寸（瀏覽器先預留空間）
<img src="photo.jpg" width="800" height="600">
Tools · 診斷工具

怎麼找出問題？

必備工具
  • Chrome DevTools Performance錄製一段時間的所有活動，看 timeline
  • Network 面板看每個資源的 waterfall（瀑布圖）
  • Lighthouse內建在 DevTools，產生效能報告 + 優化建議
  • WebPageTest第三方測試，可選不同地區、不同網速
  • PageSpeed InsightsGoogle 官方工具，整合 Core Web Vitals
  • RUMReal User Monitoring，蒐集真實使用者數據（如 Sentry、Datadog）

實務經驗：先用 Lighthouse 跑一輪，它會列出最影響分數的問題。 逐一解決後再用 Performance 面板看細節。最後上線後用 RUM 監控真實使用者體驗—— 實驗室數據常常比真實使用者快很多（因為真實使用者用爛網路、舊手機）。

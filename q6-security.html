Q · 06 · Web Security

網站被攻擊，
到底是哪個環節破洞？

你做財務應用（AlphaTrader）對資安特別敏感——一旦出事不只丟臉，是真實的金錢損失。 這一章把五大攻擊面逐一拆開：每種攻擊怎麼運作、怎麼防、為什麼某些防禦看起來瑣碎卻必要。

Attack Surface · 攻擊面總覽

網站被駭，
大概是這五種情境之一

OWASP（開放網頁應用程式安全計畫）每幾年發布 Top 10 漏洞清單。下面是 Web 工程師最該認識的五類攻擊。

▸ 五大攻擊面分布
① XSS · 跨站腳本

攻擊者把惡意 JS
塞進你的網頁

XSSCross-Site Scripting （Cross-Site Scripting）的核心：攻擊者讓自己寫的 JavaScript 跑在受害者的瀏覽器上， 可以偷 Cookie（拿到登入身份）、修改畫面、誘導點擊。

▸ Stored XSS 攻擊流程
XSS 三種類型
  • Stored XSS惡意內容存進資料庫，每個看到的人都中招（最危險）
  • Reflected XSS惡意內容在 URL 裡，受害者點連結才中招
  • DOM-based XSS純前端漏洞，JS 把使用者輸入當作 HTML 處理
防禦方法
  • Output Encoding顯示使用者內容前先 escape（< 變成 &lt;）
  • HttpOnly Cookie讓 JS 拿不到 Cookie，即使 XSS 也偷不到
  • CSPContent Security Policy，限制可執行的 script 來源
  • 用框架React、Vue 預設會 escape 變數內容
② CSRF · 跨站請求偽造

趁你還登入著，
偷偷以你的名義送請求

CSRFCross-Site Request Forgery 利用一個事實：你在 A 網站登入後，瀏覽器會把 A 的 Cookie自動帶到所有 A 網站的請求， 不管請求是從哪個網站發起的。

▸ CSRF 攻擊流程
防禦方法
  • SameSite Cookie設 Strict / Lax，跨站請求不帶 Cookie（最簡單）
  • CSRF Token每個表單塞一個隨機 token，攻擊者拿不到
  • Referer 檢查確認請求從同源網站發起
  • 重要操作 2FA轉帳、改密碼這種敏感動作要再驗證一次
③ SQL Injection · SQL 注入

把使用者輸入
當SQL 指令執行了

經典老攻擊，但 2024 還是常見。原因：開發者把使用者輸入直接拼進 SQL 字串裡。

# ❌ 錯誤寫法（字串拼接）
sql = "SELECT * FROM users WHERE name = '" + user_input + "'";

# 正常情況：user_input = "KS"
SELECT * FROM users WHERE name = 'KS';

# 攻擊者輸入：' OR '1'='1
SELECT * FROM users WHERE name = '' OR '1'='1';
# → 條件永真，撈出全部使用者

# 更慘的：'; DROP TABLE users; --
SELECT * FROM users WHERE name = ''; DROP TABLE users; --';
# → 整個 users 表被刪
防禦方法
  • Parameterized Query用佔位符（?, $1）讓資料庫自己處理引號 escape
  • 用 ORM主流 ORM 預設都用 prepared statement
  • 最小權限應用伺服器連資料庫的帳號不要給 DROP 權限
  • WAFWeb Application Firewall 過濾常見攻擊 pattern
# ✓ 正確寫法（參數化查詢）
db.query("SELECT * FROM users WHERE name = ?", [user_input]);

# 即使 user_input 是惡意字串，也只會被當作「字串值」
# 不會被當成 SQL 語法的一部分
④ DDoS · 分散式阻斷服務

用暴力流量把網站灌爆

DDoSDistributed Denial of Service 不是「駭進系統」，而是「讓系統無法服務正常使用者」。 攻擊者控制成千上萬台被感染的電腦（殭屍網路 botnet），同時發送請求把伺服器打爆。

攻擊層級 原理 規模 防禦方
L3/L4 · 網路層 SYN flood、UDP flood，用大量無效封包塞滿頻寬 Tbps 等級 CDN / Anti-DDoS 服務
L7 · 應用層 大量看似正常的 HTTP 請求，但專挑昂貴 API（搜尋、報表） 難偵測 WAF + Rate Limiting
慢速攻擊 建立連線後刻意慢慢傳資料，佔用 socket 不放 少量主機就夠 連線 timeout 設定
防禦方法
  • CDNCloudflare、Akamai 都有大量頻寬可吸收攻擊
  • Rate Limiting每個 IP 每秒最多 N 次請求
  • CAPTCHA判斷是真人還是機器
  • Auto-scaling流量大時自動加伺服器（成本會很高）
  • 地理封鎖不服務的地區直接擋掉
⑤ Supply Chain · 供應鏈攻擊

你用的第三方套件被植入後門

現代網站使用幾百個 npm 套件，每個套件又依賴幾十個其他套件——一個 React 專案可能間接用到上千個套件。 只要其中任何一個被滲透，整個網站就有風險。這是近年成長最快的攻擊類型。

經典案例：2024 年的 XZ Utils 後門事件—— 一個攻擊者花了兩年慢慢進入某個 Linux 工具的維護團隊，最後植入後門，差點影響全球幾乎所有 Linux 系統。 Web 領域的 npm 也發生過多次：event-stream、colors、node-ipc 等知名套件都被滲透過。

防禦方法
  • 鎖版本用 package-lock.json / yarn.lock 鎖住所有依賴版本
  • 定期掃描npm audit、Snyk、Dependabot 自動檢查已知漏洞
  • SBOMSoftware Bill of Materials，記錄所有依賴清單
  • 最小化依賴能不裝就不裝，能用原生 API 就別裝套件
  • Subresource Integrity從 CDN 載入 JS 時驗證 hash

這對你做 AlphaTrader 特別重要——金融應用一旦被植入後門，可能不只資料外洩，還會被竄改交易邏輯。 建議建立獨立的依賴審查流程，新加套件要看下載量、維護活躍度、過去是否出過事件。

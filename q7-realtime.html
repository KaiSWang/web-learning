Q · 07 · Real-time Technologies

聊天訊息秒到、
股價跳動，
怎麼做到的？

傳統 HTTP 是請求-回應模型：客戶端問才有答。 但聊天、即時報價、線上遊戲需要伺服器主動推送給客戶端—— 這就是即時通訊的需求。Web 演進出四種解法。

The Problem · HTTP 的限制

傳統 HTTP 不能主動推

HTTP 設計上是「客戶端發起，伺服器回應」。如果伺服器有新訊息，沒辦法主動通知客戶端—— 除非客戶端持續發問（輪詢 polling）。

▸ 四種即時技術的時序對比
Comparison · 詳細比較

四種技術的選擇邏輯

①

Long Polling · 長輪詢

最早的方案，現在很少新專案用

客戶端發請求，伺服器故意不立刻回，「掛著」直到有訊息才回應。 客戶端收到後馬上再發一個新請求。優點：相容性好（普通 HTTP 就行）； 缺點：每次訊息都要重新建立連線、效率低、伺服器要維護大量等待連線。

適用場景：舊系統、必須繞過嚴格防火牆、無法升級的環境。新專案幾乎不選這個。

②

SSE · Server-Sent Events

單向推送，比 WebSocket 簡單

建立一個持續開啟的 HTTP 連線，伺服器可以源源不斷推訊息給客戶端。 但這條連線只能單向（server → client），客戶端要送資料還是得用普通 HTTP。

# 客戶端（瀏覽器原生支援）
const es = new EventSource('/api/stream');
es.onmessage = (e) => console.log(e.data);

# 伺服器（每個 message 都是文字格式）
data: 新訊息 1

data: 新訊息 2
適合場景
  • 新聞推播單向更新，不需要客戶端回話
  • 即時股價數值持續更新（你做 AlphaTrader 可以考慮）
  • 進度通知長時間任務的進度條
  • AI 串流回應ChatGPT 那種逐字輸出

SSE 用文字協定（明文），不像 WebSocket 是 binary frame。 自動重連、訊息 ID 標記都內建在標準裡，比 WebSocket 簡單得多。

③

WebSocket · 全雙工通訊

即時通訊的「標準答案」

從 HTTP 升級而來，建立一條雙向永久通道，兩邊隨時都能發訊息。 是聊天、線上遊戲、協作編輯的主流選擇。

# 客戶端
const ws = new WebSocket('wss://example.com/chat');
ws.onopen = () => ws.send('Hello');
ws.onmessage = (e) => console.log(e.data);

# 伺服器（Node.js + ws 函式庫）
wss.on('connection', (ws) => {
  ws.on('message', (msg) => {
    // 廣播給所有人
    wss.clients.forEach(c => c.send(msg));
  });
});
適合場景
  • 聊天室LINE、Slack、Discord
  • 多人線上遊戲需要極低延遲
  • 協作編輯Notion、Google Docs、Figma
  • 即時看板Trello 拖曳卡片即時同步
  • 金融交易下單、撤單、即時行情
④

WebRTC · 點對點直連

視訊通話、檔案傳輸

前面三種都是「客戶端 ↔ 伺服器」，WebRTC 是「客戶端 ↔ 客戶端」。 適合大量資料（影像、音訊）——如果走伺服器，伺服器頻寬會爆炸。

但 NAT、防火牆會擋直連。WebRTC 用 STUN / TURN 伺服器幫兩邊「打洞」建立連線。 連不通時 fallback 到 TURN 中繼。

適合場景
  • 視訊會議Google Meet、Zoom Web 版底層
  • P2P 檔案分享WeTransfer 之類網頁版
  • 遠端桌面瀏覽器內遠端操作
  • 線上多人遊戲需要超低延遲又不想花太多伺服器費用
技術 方向 連線數 典型用途 難度
Long Polling 單向 每訊息 1 條 舊系統相容 低
SSE 單向（→ client） 1 條持久 新聞、股價、AI 串流 低
WebSocket 雙向 1 條持久 聊天、遊戲、協作 中
WebRTC 雙向（P2P） 不經伺服器 視訊、檔案 高
